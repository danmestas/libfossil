# libfossil docs site + landing page + Cloudflare auto-deploy

**Date:** 2026-04-25
**Branch:** `feat/docs-site-cf-deploy`
**Status:** Design — awaiting implementation plan

## Goal

Add a public web presence for libfossil that mirrors the sibling project [`dagnats`](https://github.com/danmestas/dagnats):

- A single-page landing site at the repo root (`site/index.html`).
- A documentation site built with Hugo + Hextra (`docs/site/`).
- Auto-generated Go API reference via `gomarkdoc`.
- Two `llms.txt`-style artifacts for LLM consumption.
- Cloudflare Workers Static Assets deployment, auto-deployed on every push to `main` via Cloudflare's git integration (no GitHub Actions deploy step).

## Non-goals

- No versioned docs (single live version follows `main`).
- No analytics, no comments, no auth.
- No GitHub Action for deployment — Cloudflare's git integration handles it.
- No changes to the existing `.github/workflows/test.yml` test pipeline.

## Visual aesthetic

The dagnats landing page sets the visual vocabulary, and libfossil reuses it verbatim. The look is a technical-manual / spec-sheet aesthetic, not a colored brand palette:

- **Background:** near-black (`--ink: #0a0a0b`).
- **Foreground:** chalk/cream (`--chalk: #e8e6e1`, `--paper: #f5f3ee`).
- **Body type:** JetBrains Mono (`--mono`) — drives nav, metadata table, code, labels.
- **Display type:** Instrument Serif italic (`--serif`) — used for the wordmark and section titles.
- **Accent (visible at the top of the page):** ember orange (`--ember: #ff4f1a`) — install-block `$` prompt, primary CTA hover, italic emphasis in headings, comparison column highlight, link selection, syntax keywords.
- **Secondary accent (lower on the page):** teal (`--teal: #00d4aa`) — architecture-diagram nodes, "us" column of the problem grid, function-name syntax highlighting, a few feature icons.
- **Layout idioms:** §01/§02 section numbering, right-aligned section eyebrow labels, hairline rules, dotted-border metadata "spec sheet" table, boxed code blocks with `SHELL`/`COPY` chrome.

Implementation reuses the dagnats CSS and HTML chrome verbatim — only the content underneath changes. The spec does not redefine the visual system.

## Component 2 — Docs site (`docs/site/`)

Hugo (extended) + Hextra theme via Hugo modules. Mirrors dagnats's `hugo.yaml` with three deltas:

- `title: libfossil`
- GitHub menu URL → `https://github.com/danmestas/libfossil`
- `editURL.base` → `https://github.com/danmestas/libfossil/edit/main/docs/site/content`
- Menu adds an `SDK` entry pointing to `/docs/reference/sdk/`

Inherited from Hextra without modification: dark/light toggle (system default), edit-on-GitHub link, wide page width, FlexSearch (client-side, indexes `content`, forward tokenization).

**Reading order** (Hextra menu weights): Quickstart → Architecture → Migration from Fossil → Extension points → Testing → SDK → Examples.

## Component 3 — Auto-generated SDK reference

`gomarkdoc` reads Go doc comments and writes Hextra-compatible markdown. Output is committed to the repo (so GitHub renders the pages and search engines index them); a pre-commit hook re-runs the generator and fails the commit if the working tree differs afterward, keeping committed pages in sync with the code.

`make docs-gen-sdk`:

```make
docs-gen-sdk:
	@command -v gomarkdoc >/dev/null || go install github.com/princjef/gomarkdoc/cmd/gomarkdoc@latest
	gomarkdoc --output docs/site/content/docs/reference/sdk/libfossil/api.md         ./
	gomarkdoc --output docs/site/content/docs/reference/sdk/cli/api.md               ./cli/
	gomarkdoc --output docs/site/content/docs/reference/sdk/db/api.md                ./db/
	gomarkdoc --output docs/site/content/docs/reference/sdk/db/driver/modernc/api.md ./db/driver/modernc/
	gomarkdoc --output docs/site/content/docs/reference/sdk/db/driver/ncruces/api.md ./db/driver/ncruces/
	gomarkdoc --output docs/site/content/docs/reference/sdk/observer/otel/api.md     ./observer/otel/
	gomarkdoc --output docs/site/content/docs/reference/sdk/dst/api.md               ./dst/
```

Excluded: `internal/` (unexported, not part of the public API surface).

## Component 4 — `scripts/gen-llms-txt.sh`

Two artifacts written into `docs/site/static/`, served by Hugo at the site root:

- **`/llms.txt`** — curated summary written inline as a heredoc. Sections: project blurb, Key Concepts (Repo, Transport, Driver, Observer, DST), Quick Start (4 steps), Go SDK Packages (7 entries), Documentation pointer.
- **`/llms-full.txt`** — generated by walking `docs/site/content/**/*.md`, stripping YAML frontmatter, concatenating with `# Source: <path>` separators, prefixed with a UTC generation timestamp.

Both files are gitignored (build artifacts only). Invoked by `make docs-gen-llms` and again by the inline build in `wrangler.jsonc`, so the deployed site is always fresh.

Curated `llms.txt` content:

```
# libfossil

> Pure-Go library and CLI for Fossil SCM repositories.
> Zero CGo. Static binaries. WASM-ready.

## Key Concepts
- Repo — open .fossil SQLite-backed repository handle
- Transport — pluggable sync interface (HTTP, custom)
- Driver — pluggable SQLite implementation (modernc, ncruces)
- Observer — hooks for sync/checkout/commit (OpenTelemetry-ready)
- DST — deterministic simulation harness for fault injection

## Quick Start
1. go install github.com/danmestas/libfossil/cmd/libfossil@latest
2. libfossil clone <url> repo.fossil
3. Open from Go: repo, err := libfossil.Open("repo.fossil")
4. Sync: repo.Sync(ctx, transport)

## Go SDK Packages
- libfossil — Repo handle, Open/Create, public API
- cli/ — Kong command structs (embeddable)
- db/ — SQLite abstraction with pluggable drivers
- db/driver/modernc — pure-Go SQLite (default)
- db/driver/ncruces — Wasm-friendly SQLite
- observer/otel — OpenTelemetry integration
- dst/ — deterministic simulation harness

## Documentation
Full docs: <site URL — set once domain is configured>
```

## Component 5 — Makefile additions

| Target | Action |
|---|---|
| `docs-serve` | `cd docs/site && hugo server -D` (local preview with drafts) |
| `docs-build` | `cd docs/site && hugo --minify` (production build) |
| `docs-gen-sdk` | as above |
| `docs-gen-llms` | `bash scripts/gen-llms-txt.sh` |
| `docs` | runs `docs-gen-sdk` → `docs-gen-llms` → `docs-build`, in order |

The `docs-gen-sdk` target is also wired into the existing `setup-hooks` pre-commit hook with a working-tree-drift check.

## Component 6 — Cloudflare deploy (`wrangler.jsonc`)

Top-level `wrangler.jsonc`:

```jsonc
{
  "$schema": "node_modules/wrangler/config-schema.json",
  "name": "libfossil-docs",
  "compatibility_date": "2026-04-25",
  "observability": { "enabled": true },
  "assets": { "directory": "./docs/site/public" },
  "build": {
    "command": "curl -sSL https://github.com/gohugoio/hugo/releases/download/v0.160.0/hugo_extended_0.160.0_linux-amd64.tar.gz | tar xz hugo && go install github.com/princjef/gomarkdoc/cmd/gomarkdoc@latest && make docs-gen-sdk && bash scripts/gen-llms-txt.sh && cd docs/site && ../../hugo --minify"
  },
  "compatibility_flags": ["nodejs_compat"]
}
```

Build steps, in order:

1. Download pinned Hugo extended (`v0.160.0` — same pin as dagnats).
2. Install `gomarkdoc` (CF build env has Go preinstalled).
3. `make docs-gen-sdk` — refresh SDK pages from the current code.
4. `bash scripts/gen-llms-txt.sh` — refresh both llms artifacts.
5. `hugo --minify` — write static output to `docs/site/public/`.
6. Wrangler serves `docs/site/public/` via Workers Static Assets.

This regenerates SDK pages on every CF build (dagnats does not — it relies on the committed pages being in sync). The cost is ~10s extra build time; the benefit is the deployed site can never drift from current code even if a contributor forgets the pre-commit hook.

## Component 7 — Auto-deploy mechanism

Cloudflare Workers' git integration handles auto-deploy. No GitHub Actions deploy step.

**One-time operational steps** (the user performs these in the CF dashboard):

1. Workers & Pages → Create → Connect to Git → select `danmestas/libfossil`.
2. Production branch = `main`. Build command and output dir come from `wrangler.jsonc`.
3. Default URL: `libfossil-docs.<cf-subdomain>.workers.dev`. Custom domain attached later.
4. Preview-branch deploys: enable for PRs (free, useful for reviewing doc changes).

The existing `.github/workflows/test.yml` is unchanged. Tests and docs-deploy are independently triggered.

## Open items not blocking design

- **Custom domain** — attached later via the CF dashboard; not required for v1.
- **Content prose** — drafted during implementation from `README.md` and existing `docs/*.md`; user iterates.
- **`baseURL` in `hugo.yaml`** — left empty in v1 (matches dagnats); set once a custom domain is attached.

## Out of scope (deferred)

- Analytics / comments / auth.
- A GitHub Action for deployment.
- Search beyond FlexSearch.

## Testing strategy

Manual verification once implemented:

1. `make docs-serve` renders the site locally with the SDK pages and llms artifacts present.
2. `make docs` produces `docs/site/public/` with all pages, search index, and `static/llms*.txt` artifacts.
3. The pre-commit hook fails when SDK pages are out of date and passes when they are in sync.
4. CF deploy succeeds and the live URL serves the landing page at `/` and the docs at `/docs/`.
